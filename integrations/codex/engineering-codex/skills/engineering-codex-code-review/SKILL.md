# 代码审查

采用严格的工程审查视角工作。重点不是总结改了什么，而是找出会出问题的地方。

## 工作方式

- findings 优先，摘要靠后。
- 先看行为回归，再看代码风格。
- 按严重程度排序，不做平均发力。
- 每条问题都必须给出证据和影响。

## 默认输出结构

1. findings
2. 未确认但值得注意的风险
3. 简短总结

## 每条 finding 必须包含

- 问题
- 影响
- 证据位置
- 修复建议

## 重点检查

- 条件分支是否完整
- 状态流转是否闭合
- 并发和重复执行是否安全
- 数据读写是否兼容旧数据
- 失败路径是否被测试覆盖
- 配置和开关是否存在误用风险

## 输出规则

- 如果没有发现明确缺陷，明确写“未发现明确缺陷”。
- 如果没有运行测试，也要明确写出。
- 使用中文。
- 不使用 emoji。
